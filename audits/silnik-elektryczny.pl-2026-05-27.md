## P0 — Critical (fix this week)

### [LIVE] robots.txt malformed — first line is `roUser-agent: *`, breaks all UA-group rules
**Where:** `frontend/public/robots.txt:1` (source) and live `https://www.silnik-elektryczny.pl/robots.txt`.
**Evidence:**
```
$ curl -s https://www.silnik-elektryczny.pl/robots.txt
roUser-agent: *
Allow: /

Disallow: /checkout
Disallow: /zamowienie/

Sitemap: https://www.silnik-elektryczny.pl/sitemap-index.xml
```
PSI Lighthouse audit (mobile, product page) confirms: `robots-txt — score 0 — robots.txt is not valid`. Per Google's robots.txt spec a rule (`Allow`/`Disallow`) must be preceded by a valid `User-agent:` line; the literal token `roUser-agent` is not a known directive, so Google parses the file as having **no UA group**, which makes all four rules (`Allow: /`, both `Disallow:` lines) orphaned and ignored. The standalone `Sitemap:` directive is the only line that still works.
**Impact:** `/checkout/` and `/zamowienie/*` are not actually blocked from crawl. /checkout has `<meta name="robots" content="noindex">` so it won't be indexed, but it still wastes crawl budget; `/zamowienie/sukces/` and `/zamowienie/anulowane/` have NO noindex (see P1 #5) and are therefore crawlable AND indexable. Also: PSI/Lighthouse SEO score drops from a perfect 100 to 92 specifically due to this audit — the same signal goes to other SEO tools and to Search Console robots.txt tester.
**Fix:** Edit `frontend/public/robots.txt:1` — change the literal first byte sequence `roUser-agent: *` to `User-agent: *`. Diff:
```diff
-roUser-agent: *
+User-agent: *
 Allow: /
```
Then `cd frontend && ./deploy.sh`. Re-verify with `curl -s https://www.silnik-elektryczny.pl/robots.txt | head -1` (must read `User-agent: *`), then GSC → robots.txt Tester → submit.

### [LIVE] Custom 404 page never served — S3/CloudFront error-doc misconfig
**Where:** CloudFront distribution `EAW287H7GJ9N6` Custom Error Response config; Astro builds `dist/404.html`; bucket has no `404/index.html` at the path CloudFront/S3 looks for.
**Evidence:**
```
$ curl -sIL https://www.silnik-elektryczny.pl/this-page-does-not-exist/
HTTP/1.1 404 Not Found
x-amz-error-code: NoSuchKey
x-amz-error-message: The specified key does not exist.
x-amz-error-detail-Key: 404/index.html
```
S3 returns 404 with `NoSuchKey` and the bare XML S3 error payload — not the Astro-designed 404 page that exists at `dist/404.html` (singular HTML file, not under `404/` directory). Users and crawlers hitting any non-existent URL get a raw S3 error page instead of the branded site 404.
**Impact:** Bad UX on every dead link (e.g. links from old GSC URLs that no longer exist, mistyped product slugs, link-rot from external sites). Branding gap. Hurts trust signals. Status code is correct (404 — Google does process it as 404), so this is not an indexation bug but it is a P0 because it affects every error page on the site.
**Fix:** Two options — pick one and apply:
1. **(simpler) Update CloudFront Error Response**: in distribution `EAW287H7GJ9N6` → Error pages → 404 → set Response Page Path to `/404.html` (not `/404/index.html`) and Response Code 404. Then `aws cloudfront create-invalidation --distribution-id EAW287H7GJ9N6 --paths "/*"`.
2. **(rebuild path) Generate `dist/404/index.html`** by adding `trailingSlash: 'always'` or a manual `src/pages/404/index.astro` so the path on disk matches what CloudFront looks for.

### [LIVE] `Astro.redirect('/404')` in product page uses default 302 status (should be 404 or 410)
**Where:** `frontend/src/pages/silnik/[slug].astro:13`:
```ts
if (!data?.success) return Astro.redirect('/404');
```
**Evidence:** Astro `redirect()` defaults to 302. For invalid product slugs the user/crawler gets `302 Found → /404 → 404 NoSuchKey`. Even ignoring the P0 #2 above, the 302 chain is wrong semantics — a missing product is *not* "this resource is temporarily found at another URL", it's "this resource doesn't exist".
**Impact:** Soft-404 risk: Googlebot follows the 302, processes the chain, and may delay or fail to deindex stale product URLs. Mixed signals slow GSC's "Removed from index" verdict on discontinued products. This is the Astro-redirect-default-302 pattern previously seen in matury-online.pl.
**Fix:** Static Astro can't return a 404 status from a page handler (output is HTML written to disk). Two real options:
1. Replace `Astro.redirect('/404')` with `Astro.rewrite('/404')` (Astro 4.13+, supported in 5.x) — serves 404 body without changing URL. Still 200 status on disk, but no 302 hop.
2. Better: at build time, only generate paths from `getAllMotorSlugs()` (line 7) and **drop the runtime check entirely** since `getStaticPaths` already constrains to valid slugs. The check at line 12 is unreachable in static output unless the API changes between `getStaticPaths` and `getMotor` calls; remove it.

### [LIVE] Trailing-slash mismatch: canonical points to no-slash URL, live serves at /-slash, redirect is 302
**Where:** `frontend/src/pages/kategoria/[slug].astro:78` sets `canonical={`https://www.silnik-elektryczny.pl/kategoria/${slug}`}` — without trailing slash. `frontend/src/pages/silnik/[slug].astro:76` similarly: `const canonical = `https://www.silnik-elektryczny.pl/silnik/${slug}`` — without slash. `frontend/src/layouts/Layout.astro:16-20` NAV_CATS all use no-slash hrefs (`/kategoria/trojfazowe`). But live S3 serves the page at the **slashed** URL and 302-redirects the bare URL.
**Evidence:**
```
$ curl -sIL https://www.silnik-elektryczny.pl/kategoria/trojfazowe
HTTP/1.1 302 Moved Temporarily
Location: /kategoria/trojfazowe/
HTTP/1.1 200 OK
$ curl -s https://www.silnik-elektryczny.pl/kategoria/trojfazowe/ | grep -oE '<link rel="canonical"[^>]*>'
<link rel="canonical" href="https://www.silnik-elektryczny.pl/kategoria/trojfazowe">
```
So the page accessible at `…/trojfazowe/` declares its canonical as `…/trojfazowe` — which itself 302-redirects back to `…/trojfazowe/`. Google has to figure out which form is the "real" page; click-paths from the site nav always cost a redirect; PageRank dilutes across the 302.
**Impact:** Canonical signal is self-contradicting on every category and every product page. Google generally tolerates this but will (a) keep both forms in its index briefly, (b) wait longer to consolidate, (c) treat 302 as non-permanent, so the slash form never becomes canonical with certainty. On a low-DA (10) site like this every signal counts.
**Fix:** Pick one and be consistent. Astro default is no-trailing-slash, but the live S3 config (`/path` → 302 → `/path/`) implies the deployed convention is *with*. Easiest fix:
1. In `astro.config.mjs` set `trailingSlash: 'always'` — Astro will then emit canonicals with trailing slash, and the 302 will no longer happen on internal links if those nav hrefs are also updated.
2. Update `frontend/src/layouts/Layout.astro:16-20,146-147` NAV_CATS to use trailing slashes (`/kategoria/trojfazowe/`).
3. Also update `frontend/src/pages/index.astro` cat-tile/pow-tile hrefs (lines 184, 201) to trailing-slash variants.
4. Switch CloudFront to **301** (not 302) for the slash-normalization redirect, so even external bare-slash links pass link equity. CloudFront Functions or a Lambda@Edge `if (uri && !uri.endsWith('/') && !uri.includes('.')) return 301`.
